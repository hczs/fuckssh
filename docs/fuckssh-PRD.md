# fuckssh 产品需求文档 (PRD)

## 文档信息

- 版本：v1.0
- 日期：2026-05-22
- 作者：产品负责人

## 1. 产品概述

### 1.1 产品背景

个人开发者与公司程序员在管理多台 VPS（约 3～60 台）时，依赖标准 OpenSSH 生态（`~/.ssh/config`、本地密钥、终端与 VS Code Remote SSH）。当前常见做法是使用 Tabby 等客户端在各台电脑上分别配置，与命令行 `ssh` 不互通；新机器上架时需自行搜索教程，完成密钥生成、公钥上传、`ssh config` 编写等步骤，心智负担高、易出错。

fuckssh 是一款跨平台 CLI，以本机 `~/.ssh/config` 为唯一配置来源，不引入私有配置格式，确保用户在新电脑上即使未安装本工具，仍可继续使用标准 SSH 配置连接服务器。

### 1.2 产品目标

- 将「新 VPS 从向导到 `ssh 别名` 免密可用」压缩到 **3 分钟以内**
- 消除用户搜索 SSH 密钥与 `ssh config` 教程的需求
- 在本机稳定列出并模糊搜索所有已配置的 VPS 别名
- 保持与 Tabby、VS Code Remote SSH、系统 `ssh` 命令的互通（仅操作标准 SSH 文件）

### 1.3 目标用户

| 维度 | 描述 |
|------|------|
| 角色 | 个人开发者、公司程序员 |
| 平台 | Windows、macOS、Linux（优先级按各平台市场占有率） |
| 规模 | 每人约 3～60 台 VPS |
| 设备场景 | 常见「公司电脑 + 家里电脑」双机使用，不频繁更换整机 |
| 现状工具 | Tabby（公司与家里各配一遍，不同步）；命令行 `ssh` 需单独维护 |

## 2. 用户需求

### 2.1 核心痛点

1. **VS Code Remote SSH 配置成本高**：不熟悉命令的用户需自行完成 `ssh-keygen`、公钥部署、`ssh config` 编写，步骤分散。
2. **GUI 客户端与 CLI 不互通**：Termius、MobaXterm、Tabby 内配置的连接无法直接用于命令行 `ssh`，往往仍需输入密码。
3. **公钥与 config 认知门槛**：不清楚公钥应放在服务器何处；不清楚 `ssh config` 中私钥相关字段（如 `IdentityFile`）如何填写。
4. **双机重复劳动**：公司、家里各配置一遍，无统一同步手段（V2 规划加密备份恢复）。
5. **次要痛点**：VS Code Remote 偶发仍要求输入密码（发生概率不高，MVP 不专项优化）。

### 2.2 用户场景

**场景 A：新购云主机，密码登录可用（MVP 主路径）**

用户获得 IP/域名、用户名与初始密码，希望通过交互式向导完成：可选端口/别名/算法 → 生成密钥 → 自动部署到远端 → 写入 `ssh config` → 提示可直接执行 `ssh <别名>`。

**场景 B：本机已有可用私钥，仅缺/乱 `ssh config`（MVP 次路径）**

远端已接受该密钥登录，用户选择「密钥连接」，提供本机私钥路径，工具仅写入正确的 `Host` / `IdentityFile` 等条目（可选将密钥整理至 `~/.ssh` 目录，实现阶段再定）。

**场景 C：管理多台 VPS（MVP）**

用户需快速浏览当前所有 Host 别名，并通过别名、hostname 或 IP 模糊搜索定位目标机器。

**场景 D：公司 ↔ 家里双机同步（V2）**

用户在公司机完成配置后，通过一条命令将 `ssh config` 与私钥加密备份到可配置目录（如同步盘 OneDrive），在家解密恢复，并处理别名冲突合并。

### 2.3 现有解决方案

- 网上搜索 SSH 密钥生成教程
- 本地 `ssh-keygen` 生成密钥
- 查找公钥目录，使用 FTP 等工具将公钥上传至服务器
- 手动编辑 `ssh config`（密钥路径、别名等）
- 使用 Tabby 分别在多台电脑上重复上述流程，不做同步

## 3. 产品功能

### 3.1 核心功能 (MVP)

#### 3.1.1 新 VPS 一站式配置向导

交互式 CLI 向导，支持两种连接模式：

| 模式 | 适用条件 | 工具行为 |
|------|----------|----------|
| 密码连接 | 新购云主机，已开启密码登录（root/ubuntu） | 生成密钥 → 自动配置远端 `authorized_keys` → 写入 `ssh config` |
| 密钥连接 | 远端已能用指定私钥登录，仅缺 config | 用户提供私钥路径，工具写入 `Host` / `IdentityFile` 等 |

**输入字段：**

| 字段 | 必填 | 默认行为 |
|------|------|----------|
| IP / 域名 | 是 | — |
| 用户名 | 是 | — |
| 密码 | 是（密码连接模式） | 配置过程中用于首次登录与部署公钥 |
| 端口 | 否 | 询问用户；回车使用默认值（如 22），并给出提示 |
| 别名 | 否 | 回车使用自动生成规则（实现阶段定义，如基于 IP） |
| 密钥算法 | 否（密码连接） | 回车使用默认算法，降低心智负担 |

**服务器前提（密码连接）：**

- 刚购买的云主机
- 已开启密码登录
- 常见 root / ubuntu 用户
- **不包含**：已禁用密码且仅允许密钥、需合并他人已有 `authorized_keys` 等复杂场景

**完成后：** 明确提示用户可执行 `ssh <别名>` 进行免密连接。

#### 3.1.2 列出 VPS

从本机 `~/.ssh/config` 解析并展示所有 Host 条目，每条显示：

- 别名（Host 别名）
- HostName
- Port
- User

MVP **不提供** 编辑、删除能力。

#### 3.1.3 搜索 VPS

- 模糊搜索
- 匹配字段：别名、hostname、IP
- 任一字段命中即展示

### 3.2 增强功能 (V2)

| 功能 | 描述 |
|------|------|
| 加密备份 | 单命令将 `ssh config` + 私钥加密导出到用户可配置目录（如同步盘） |
| 加密恢复 | 从同步盘拉取并解密，合并到本机 |
| 合并策略 | 别名相同且 IP 相同 → 无冲突；别名相同且 IP 不同 → 自动重命名为 `vps1-2` 形式并提示用户 |
| 编辑 / 删除 Host | 列表中管理已有配置 |
| 双机同步目录配置 | 用户可指定备份目标路径 |

### 3.3 功能优先级

| 优先级 | 功能 |
|--------|------|
| P0 | 新 VPS 一站式配置（密码连接 + 密钥连接） |
| P0 | 列出 VPS |
| P0 | 模糊搜索 VPS |
| P1 | 加密备份 / 恢复与冲突合并 |
| P2 | 列表编辑、删除 |
| P3 | VS Code Remote 偶发密码问题专项排查（低优先级） |

## 4. 非功能需求

### 4.1 性能要求

- 新 VPS 向导全流程（用户输入完成 → 可 `ssh 别名` 免密）目标 **< 3 分钟**（含用户输入与网络延迟）
- 列出 / 搜索响应在本机 **< 1 秒**（config 规模在数十台量级）

### 4.2 安全要求

- 仅使用标准 OpenSSH 文件与机制，不引入私有配置数据库
- 密码仅用于配置过程中的首次连接，不应持久化明文密码
- V2 加密备份需使用用户提供的口令或等价机制，备份包可存放于同步盘
- 修改 `ssh config` 或密钥前应具备可恢复手段（实现阶段定义，如自动备份原文件）

### 4.3 兼容性要求

- 跨平台：Windows、macOS、Linux
- 平台特性按市场占有率优先适配与测试
- 产出配置须被系统 `ssh`、VS Code Remote SSH、Tabby 等标准客户端识别

## 5. 边界与约束

### 5.1 产品边界

**MVP 与整体均不做：**

- SSH 隧道 / 跳板机（JumpHost）管理
- 替代 Tabby、Termius、MobaXterm 的终端能力
- 以产品自有格式替代 `~/.ssh/config` 作为主数据源
- 自动修改远端 `sshd_config`
- 团队多人协作 / 权限管理

**设计原则：**

- CLI 只依赖标准 SSH 配置体系；用户换机未安装 fuckssh 时，仍可凭现有 `~/.ssh` 文件继续连接
- 不管理「以产品为中心」的额外配置文件；配置过程中的密钥生成与写入属于 OpenSSH 标准流程的一部分

### 5.2 技术约束

详见 [技术选型方案](./fuckssh-技术选型.md)。摘要：

- 主语言 **Go 1.22+**；CLI 框架 **Cobra**
- **MVP 用 Go 生成密钥**（Ed25519 默认），不依赖 `ssh-keygen`
- 日常连接依赖系统 **`ssh`**：启动/向导前检测，缺失则分平台提示安装（Windows 为可选功能）
- 密码模式公钥部署可用 **`golang.org/x/crypto/ssh`**；仅读写标准 `~/.ssh/config` 与密钥文件，无服务端与数据库

### 5.3 资源约束

- 首期以 MVP 可自用、可发布为里程碑
- V2 双机同步依赖用户自备同步盘（如 OneDrive），产品不托管用户数据

## 6. 成功指标

### 6.1 核心指标

| 指标 | 目标 |
|------|------|
| 新 VPS 配置耗时 | 从启动向导到 `ssh <别名>` 免密成功 **< 3 分钟** |
| 教程依赖 | 用户不再为密钥与 `ssh config` 搜索教程 |
| 列表可用性 | 公司或家里至少一台电脑能稳定列出全部 Host |

### 6.2 里程碑

| 阶段 | 交付物 | 验收标准 |
|------|--------|----------|
| MVP | 一站式配置 + 列表 + 搜索 | 满足上述核心指标，三平台基本可用 |
| V2 | 加密备份 / 恢复 + 合并策略 | 双机场景下配置可迁移，冲突规则符合 PRD |
| V3+ | 编辑删除、跳板机等 | 按反馈排期 |

## 7. 风险与依赖

### 7.1 潜在风险

| 风险 | 说明 | 缓解方向 |
|------|------|----------|
| `ssh config` 解析复杂度 | 用户手写 Include、通配符 Host 可能导致列表不完整 | 明确 MVP 解析范围，逐步增强 |
| 跨平台路径差异 | Windows 与 Unix 的 `~/.ssh` 路径、权限模型不同 | 技术选型阶段按平台验证 |
| 公钥部署失败 | 网络、防火墙、权限导致无法写入 `authorized_keys` | 向导内清晰报错与回滚指引 |
| 加密备份安全 | V2 备份包泄露可能导致密钥泄露 | 强加密 + 用户口令，文档安全提示 |

### 7.2 外部依赖

- 系统已安装 OpenSSH 客户端（`ssh`、`ssh-keygen` 等）
- 密码连接模式要求目标 VPS 在配置时允许密码登录
- V2 依赖用户自行提供的同步盘或共享目录
